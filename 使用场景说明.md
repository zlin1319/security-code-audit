# Security Code Audit - 使用场景说明

本文档说明 `security-code-audit` 当前版本的典型使用方式。内容已按仓库现状同步，避免引用过时目录、参数或流程。

## 1. 工具定位

这个项目当前是一个本地 SAST 工具，适合两类使用方式：

1. 对话式使用：在支持 Skills 的 Agent 环境里，通过自然语言触发扫描
2. 命令行使用：优先执行 `python -m security_code_audit`，也兼容 `scripts/audit.py` 和 `scripts/audit.sh`

它当前更适合做：
- 仓库基线扫描
- PR / MR 增量检查
- 多语言项目的快速安全审计
- 在对话里让 Agent 总结 findings、解释风险、给修复建议

它当前不属于“纯 AI 主审计”，主体仍是规则扫描加上下文修正，可选 AI 验证作为增强层。

## 2. 支持的语言与输入

### 支持的语言

| 语言 | 参数值 | 文件扩展名 | 说明 |
|------|--------|------------|------|
| Java | `java` | `.java` | 后端服务、控制器、DAO 等 |
| JavaScript | `javascript` | `.js`, `.jsx` | 前端和 Node.js |
| TypeScript | `typescript` | `.ts`, `.tsx` | 复用 JavaScript 规则 |
| Python | `python` | `.py` | Web 服务、脚本 |
| PHP | `php` | `.php` | 传统 Web 项目 |
| C# | `csharp` | `.cs` | .NET 服务 |
| Kotlin | `kotlin` | `.kt`, `.kts` | JVM 服务 |
| Go | `go` | `.go` | Go 服务 |
| 自动识别 | `other` | 上述所有扩展名 | 混合项目推荐 |

### 支持的输入

| 输入类型 | 示例 | 说明 |
|----------|------|------|
| 单文件 | `examples/java/VulnerableController.java` | 扫描单个文件 |
| 目录 | `./src` | 递归扫描目录 |
| 混合项目 | `./repo --language other` | 自动按扩展名识别 |
| PR / MR 改动 | `--git-diff-range origin/main...HEAD` | 只扫改动范围 |

### 暂不支持

- 压缩包直接扫描
- 二进制文件扫描
- 远程仓库直接扫描

远程仓库需要先 clone 到本地。

## 3. 输出内容

默认输出目录是 `reports/<scan-name>/`，常见产物包括：

- `audit-report.json`：机器可读结果
- `audit-report.md`：人工阅读报告
- `audit-report.sarif`：GitHub / Azure / 其他 code scanning 平台上传
- `audit-report-ai.md`：启用 `--use-ai` 或 `--skill-mode` 时的附加分析报告
- `ai-analysis-prompt.txt`：启用 `--auto-ai` 或 `--skill-mode` 时生成的分析提示

`reports/` 是运行时目录，默认不建议入库。

## 4. 场景一：对话式使用

### 适用场景

- 客户在 Claude Code / Codex 这类支持 Skills 的 Agent 环境里写代码
- 希望通过自然语言直接发起扫描
- 希望扫描后继续追问风险、误报、修复建议

### 典型话术

```text
用 security-code-audit skill 扫描这个仓库
用 security-code-audit skill 扫描 ./service，语言 go
用 security-code-audit skill 只检查当前 PR/MR 改动
用 security-code-audit skill 按项目配置扫描
```

### 底层推荐执行方式

对话式宿主应优先使用：

```bash
./scripts/skill_scan.sh --path ./src --language other
```

只有在调试 wrapper 或直接走底层 CLI 时，再使用：

```bash
python -m security_code_audit --skill-mode --path ./src --language other
```

wrapper 背后实际调用的是 `--skill-mode`，它会自动启用 AI 相关产物，并把“发现漏洞”与“执行失败”区分开：

- 扫描一次即可，不需要因为 `critical` / `high` 再重跑
- 自动生成 `audit-report-ai.md`
- 自动生成 `ai-analysis-prompt.txt`
- 即使发现高危漏洞，也返回退出码 `0`，避免对话式宿主把结果误显示为失败

### 适合做什么

- 首次接入时做全库基线扫描
- 日常开发时只检查当前改动
- 在对话里继续问：
  - 哪些是高危
  - 哪些可能是误报
  - 给出修复建议

## 5. 场景二：本地命令行使用

### 全库扫描

```bash
python -m security_code_audit --path ./src --language java
```

### 指定规则集和阈值

```bash
python -m security_code_audit \
  --path ./src \
  --language go \
  --ruleset all \
  --confidence medium
```

### 使用 shell 包装脚本

```bash
./scripts/audit.sh ./src go all ./reports/security-audit zh medium
```

### Skill 模式单次扫描

```bash
./scripts/skill_scan.sh --path ./src --language go
```

### 适用场景

- 本地自测
- 发布前检查
- 安全培训演示
- 没有 Agent Skill 宿主时直接落地

## 6. 场景三：PR / MR 增量扫描

这是当前最推荐的日常使用方式。

### 用 Git diff 范围

```bash
python -m security_code_audit \
  --path ./src \
  --language other \
  --git-diff-range origin/main...HEAD
```

### 用显式文件列表

```bash
python -m security_code_audit \
  --path ./src \
  --language other \
  --changed-files src/api/user.go src/web/login.ts
```

### 从文件读取变更列表

```bash
git diff --name-only origin/main...HEAD > .changed-files.txt

python -m security_code_audit \
  --path ./src \
  --language other \
  --changed-files-file .changed-files.txt
```

### 适用场景

- 大仓库日常检查
- PR / MR review
- 只想关注本次提交带来的安全风险

## 7. 场景四：按项目配置扫描

当团队希望统一规则、路径排除和阈值时，推荐使用项目配置文件。

支持自动发现：

- `.security-audit.toml`
- `.security-audit.json`
- `.security-audit.yaml`
- `.security-audit.yml`

### 示例

```bash
./scripts/skill_scan.sh --config .security-audit.toml
```

### 适用场景

- 团队统一规范
- 降低命令行复杂度
- 本地和 CI 统一配置

## 8. 场景五：忽略与抑制

真实项目里通常需要排除生成代码、测试目录或已接受风险。

### 路径排除

```bash
python -m security_code_audit \
  --path ./src \
  --language other \
  --exclude 'dist/*' 'vendor/*'
```

### 从 ignore 文件读取

```bash
python -m security_code_audit \
  --path ./src \
  --language other \
  --ignore-file .security-audit.ignore
```

### 行内 suppress

```php
// security-code-audit: ignore sqli-001
$query = "SELECT * FROM users WHERE name = '" . $name . "'";
```

### 适用场景

- 排除第三方代码或生成代码
- 压低已知误报
- 对特定规则做局部豁免

## 9. 场景六：CI / Code Scanning 接入

虽然这个 skill 主要也可以在对话里使用，但同一套输出也可以接到 CI。

### GitHub / GitLab / Azure

仓库已提供模板：

- `assets/ci/github-code-scanning.yml`
- `assets/ci/gitlab-ci.yml`
- `assets/ci/azure-pipelines.yml`

### 典型命令

```bash
python -m security_code_audit \
  --path ./src \
  --language other \
  --git-diff-range origin/main...HEAD \
  --output ./reports/security-audit
```

### 退出码

| 退出码 | 含义 | 建议 |
|--------|------|------|
| `0` | 无 High / Critical | 通过 |
| `1` | 有 High | 视策略决定是否阻断 |
| `2` | 有 Critical | 阻断 |

### 注意

- GitHub / Azure 可以直接消费 SARIF
- GitLab 当前模板以 SARIF artifact 为主，不等同于原生 Security Dashboard

## 10. 场景七：企业版扫描

当前代码仍支持通过 `--api-key` 进入企业增强模式。

### 命令

```bash
python -m security_code_audit \
  --path ./src \
  --language java \
```

### 适用场景

## 12. 推荐组合

### 日常开发

1. 在对话里触发 `security-code-audit` skill 做改动检查
2. 对高风险 findings 继续追问修复建议

### 团队协作

1. 仓库里放 `.security-audit.toml`
2. PR / MR 默认只扫改动
3. 定期做一次全库基线扫描

### 发布前

1. 跑全库扫描
2. 使用 `top25` 或 `owasp`
3. 使用 `--confidence high`
4. 审核 `audit-report.md` 和 `audit-report.sarif`

## 13. 常见问题

### Q: 扫描太慢怎么办？

优先做这几件事：

1. 用 `--git-diff-range` 或 `--changed-files-file` 做增量扫描
2. 用 `--ruleset top25` 缩小规则范围
3. 用 `--exclude` 排除无关目录

### Q: 误报太多怎么办？

1. 提高 `--confidence`，例如 `high`
2. 用 `--exclude` / `--ignore-file` 排除噪音目录
3. 对局部已确认误报使用行内 suppress
4. 在对话里继续让 Agent 帮你复核 findings

### Q: 如何在对话里使用？

前提是运行环境支持 Skills、本地文件访问和工具调用。常见问法：

```text
用 security-code-audit skill 扫描这个仓库
用 security-code-audit skill 只检查当前 PR/MR 改动
```

### Q: `tests/` 目录要不要保留？

建议保留。它是当前项目唯一的自动化 smoke test，用来防止 CLI 主流程被改坏。
